> **一句话**：专为 AI Agent 设计的 Context 数据库——用文件系统范式统一管理记忆、资源和技能，L0/L1/L2 三层按需加载，让 RAG 变得可观测、可调试。

## 是什么

OpenViking 是火山引擎开源的**Agent Context 数据库**，核心洞察是：传统 RAG 的向量存储是扁平的、不透明的，无法满足 Agent 长期运行的 context 管理需求。

OpenViking 用**文件系统范式**重新组织 Agent 所需的记忆、资源和 Skill，让 context 管理像管理本地文件一样直观。

## 核心架构

### 三层 Context 结构

| 层级 | 内容 | 加载策略 |
|------|------|---------|
| **L0** | 核心指令、关键规则 | 始终加载 |
| **L1** | 领域知识、常用记忆 | 按需加载 |
| **L2** | 历史记录、详细资源 | 检索触发 |

### 五大核心能力

| 能力 | 解决的问题 |
|------|-----------|
| **文件系统管理范式** | 统一管理碎片化的记忆/资源/Skill |
| **分层 Context 加载** | 减少 Token 消耗，降低成本 |
| **目录递归检索** | 结合目录定位 + 语义搜索，提升召回精度 |
| **可视化检索轨迹** | RAG 不再是黑盒，检索过程可观测 |
| **自动会话管理** | 自动压缩对话，提取长期记忆，越用越智能 |

## 技术要求

- Python 3.10+
- Go 1.22+（编译 AGFS 组件）
- GCC 9+ 或 Clang 11+（编译 core extensions）

## 与传统 RAG 的对比

| 维度 | 传统 RAG | OpenViking |
|------|---------|-----------|
| 存储结构 | 扁平向量库 | 文件系统层次结构 |
| 检索透明度 | 黑盒 | 可视化检索轨迹 |
| Context 分类 | 统一向量化 | 记忆/资源/Skill 分类管理 |
| Token 使用 | 全量加载或随机截断 | L0/L1/L2 按需加载 |
| 记忆演化 | 静态 | 会话后自动提取更新 |

## 适合场景

- 构建需要长期运行、持续积累上下文的 Agent
- 希望 RAG 检索过程可观测、可调试的团队
- 需要统一管理多类型 context（记忆+文档+工具）的复杂 Agent 系统
